Chat Application Implementation Report

Introduction
This report presents a comprehensive overview of the design choices, testing strategies, fault tolerance mechanisms, and AI utilization involved in the development of the multithreaded group chat application. The system was engineered to robustly handle multiple concurrent client connections, manage decentralized roles such as the group coordinator, and facilitate seamless public and private messaging over TCP sockets. 

1. Design Pattern [04 marks]

The complexity of a multithreaded networked application necessitates a highly decoupled and organized codebase. Implementing established design patterns was critical to achieving scalability and maintainability. Two distinct and challenging design patterns utilized in this implementation are the Command pattern and the Observer pattern.

1.1 Command Pattern
Implementation: 
The chat server must interpret and execute a variety of distinct operations sent by the client, ranging from broadcasting messages, sending private messages, requesting member lists, yielding ping acknowledgments, and graceful disconnects. To handle this, the Command pattern was implemented via a base 'ChatCommand' interface defining a single 'execute()' method. Concrete command classes, such as 'BroadcastCommand', 'PrivateMessageCommand', 'ListMembersCommand', and 'QuitCommand', implement this interface. A 'CommandFactory' is responsible for parsing the raw string input from the socket, identifying the protocol keyword, and instantiating the relevant command object along with its required parameters.

Justification: 
In a traditional, naive implementation, network servers often rely on massive, monolithic 'switch' or 'if-else' statements buried deep within the client-handler thread. This creates a severe bottleneck and violates the Open/Closed Principle. As the protocol grows, this centralized parsing block becomes unmaintainable and highly prone to regression bugs. The Command pattern elegantly abstracts the parsing from the execution logic. The handling thread simply invokes 'command.execute()', remaining entirely agnostic to the business logic of the operation. This encapsulation means adding a new feature—for instance, a file transfer protocol—requires only drafting a new Command class and registering its string keyword in the factory, without touching the core server thread loop.

1.2 Observer Pattern
Implementation: 
The server contains a centralized 'ServerState' object that tracks all active connections, their unique identifiers, and the current group coordinator. The Observer pattern was integrated by establishing a 'GroupEventListener' interface. When a 'ChatServer' thread is spawned for a new client, it acts as an observer by registering itself with the 'ServerState' subject. Whenever a state change occurs—such as a user joining, a user disconnecting, or a new coordinator being elected due to the previous one disconnecting—the 'ServerState' iterates through its registered listeners, synchronously pushing protocol updates (like 'MEMBER_JOINED' or 'COORDINATOR_CHANGED') to all clients.

Justification: 
Broadcasting synchronized events across multiple independent threads concurrently is inherently risky and can lead to deadlocks or dropped messages. Without the Observer pattern, each individual client thread would need to continually poll the central state for changes, wasting CPU cycles and flooding the application with unnecessary locks. The Observer pattern provides an efficient, push-based model. It inverts the dependency, allowing the central repository to instantly notify all interested parties the exact moment a membership or leadership event occurs, guaranteeing that all client GUIs remain accurately synchronized with minimal overhead.

2. JUnit Testing [04 marks]

Testing a concurrent, network-dependent application poses unique challenges, particularly regarding state isolation and asynchronous execution. The JUnit 5 suite addresses these complexities through targeted, isolated test cases. Four notably challenging unit tests include:

2.1 Coordinator Re-election Cascade Simulation
Testing the automated coordinator election required simulating a multi-node environment. The challenge lies in accurately recreating the chronological sequence of joins and subsequent abrupt disconnects. The test programmatically inserts mock clients into the 'ServerState' with specifically staggered 'joinedAt' timestamps. It then forcibly removes the current coordinator, asserting that the 'ServerState.electNewCoordinator()' logic accurately traverses the internal concurrent maps to identify the exact next-oldest member, and validates that the 'COORDINATOR_CHANGED' event is appropriately dispatched to the remaining stubs.

2.2 Mocking I/O for Socket Isolation
A core tenet of unit testing is the elimination of external dependencies, such as local network stacks or hardcoded ports. Testing the protocol string generation required bypassing the actual Java Sockets. The solution involved injecting mock 'PrintWriter' wrappers instantiated around 'StringWriter' objects directly into the 'ServerState'. This test validates complex logic—such as 'PrivateMessageCommand' isolation—by inspecting the 'StringWriter' buffers to guarantee that a broadcast message correctly propagates to all buffers, while a 'PRIVMSG' strictly targets a single buffer, all executed instantaneously in local memory without exposing a real TCP server.

2.3 Asynchronous Ping/Pong Timeout Detection
Validating the 'PingScheduler' health-check logic requires testing time-bound boundaries. Because unit tests should run rapidly, waiting for real 20-second scheduler intervals is inefficient. The test tackles this by directly invoking the timeout evaluation logic within a controlled temporal context. It artificially seeds the concurrent 'pongAcks' map with varying 'true/false' acknowledgment states and simulates the expiration of the 5-second timeout window. The test asserts that threads flagged 'false' (unresponsive) are aggressively deregistered and their sockets closed, while 'true' threads remain completely unaffected.

2.4 Singleton State Cross-Contamination Avoidance
Because the 'ServerState' and 'MessageLogger' rely on the Singleton pattern to share data globally across threads, they inherently pose a massive risk to test isolation. If Test A adds a user, Test B might fail because the state was not reset, leading to flaky test suites. The challenge was guaranteeing a pristine environment per test. This was overcome by leveraging JUnit's '@BeforeEach' and '@AfterEach' lifecycle annotations combined with Java Reflection protocols to forcefully reset the static internal variables of the Singleton instances, completely wiping the memory space and ensuring hermetic, deterministic test executions.

3. Fault Tolerance [04 marks]

In distributed systems, failures are inevitable. A robust chat server must seamlessly tolerate network disruptions, silent hardware crashes, and administrative node failures to maintain uninterrupted service for the remaining users.

3.1 Health Checking and Stale Socket Reclamation (PingScheduler)
The most common fault in TCP architectures is the "silent drop." If a client process terminates abnormally (e.g., losing Wi-Fi, power failure, killing the process via Task Manager), the TCP FIN packet is never sent. The server remains oblivious, keeping the ghost connection open indefinitely, polluting the member list and exhausting file descriptors. Fault tolerance against this is implemented via the 'PingScheduler'. Offloaded to a dedicated 'ScheduledExecutorService', the server broadcasts a 'PING' packet every 20 seconds. It mandates a strict protocol constraint: clients must echo a 'PONG' within 5 seconds. If the application detects a timeout, it assumes the socket is irrecoverably dead, administratively forces a close on the resource, and issues a 'MEMBER_LEFT' notification to the network, self-healing the state.

3.2 Coordinator Redundancy and Automatic Failover
The chat application incorporates a decentralized leadership model. The first client to connect inherently assumes the 'Coordinator' role. While the coordinator is a singular point of structural truth, the system provides rigid fault tolerance against coordinator failure. If the coordinator drops off—either gracefully via a 'QUIT' command or abruptly caught by the PingScheduler's cleanup routine—the 'ServerState' halts briefly to execute a deterministic 'electNewCoordinator()' routine. Utilizing the immutable 'joinedAt' timestamps generated upon client registration, leadership is immediately transferred to the node with the highest seniority. This transparent failover prevents any leadership vacuum, ensuring operations continue without requiring manual intervention or server reboot.

3.3 Concurrent Memory Access Defenses
Data corruption resulting from unpredictable thread interleaving represents a severe fault vector. Because multiple clients are constantly joining, leaving, and reading the membership roster concurrently, iterating across basic 'HashMap' or 'ArrayList' structures will inevitably throw a 'ConcurrentModificationException' and crash the handling thread. The implementation actively defenses against thread races by housing all core registries within 'ConcurrentHashMap' and 'CopyOnWriteArrayList' objects. These atomic data structures guarantee that reads can happen concurrently with writes across the group without risking structural corruption, ensuring high availability even under heavy, unpredictable query loads.

4. Usage of AI [04 marks]

During the development lifecycle, AI tools (such as GitHub Copilot and ChatGPT) were strategically utilized as advanced pair-programming assistants to accelerate prototyping and resolve obscure API caveats, notably in the realms of thread concurrency and repetitive text processing.

4.1 Architectural Paradigms and Code Refactoring
Initial versions of the server architecture relied heavily on a monolithic procedural loop to parse socket inputs. AI was prompted to evaluate the codebase and recommend architectural improvements based on SOLID principles. It suggested refactoring the input handler into the Command Pattern. AI assisted by scaffolding the boilerplate for the 'ChatCommand' interface and the corresponding 'CommandFactory', which significantly reduced the time spent writing repetitive class structures and allowed more focus on the internal execution logic of complex commands like 'ListMembers'.

4.2 Regular Expressions and Edge-Case Hardening
Parsing input strings securely, especially the 'PRIVMSG @targetId message_content' syntax, required precise logic. Attempting to split strings natively using 'String.split()' can be treacherous, frequently resulting in 'IndexOutOfBoundsExceptions' if a user inputs malformed syntaxes or sequential whitespaces. AI was leveraged to generate and refine robust Regular Expressions (Regex) and safe array-boundary checks to guarantee that even malicious or accidentally mangled inputs would be routed to a graceful error handler rather than crashing the server thread.

4.3 Java Concurrency Problem Solving
While implementing the Observer pattern, the system sporadically threw 'ConcurrentModificationExceptions' during heavy load testing when broadcasting messages simultaneously alongside user registration events. By querying AI with the specific stack traces and snippets of the 'ServerState' iterators, the AI helped identify the hidden race conditions. It recommended transitioning from synchronized block wrappers to inherently thread-safe primitives like 'ConcurrentHashMap', effectively resolving the deadlock risks and stabilizing the application's runtime.
